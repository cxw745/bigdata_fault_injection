# 大数据故障注入系统 - 实验目标与配置说明

## 📋 文档概述

本文档详细描述了大数据故障注入系统的实验目标和配置文件说明，旨在为大数据故障检测研究提供实验平台。

---

## 🎯 实验目标

### 1. 项目背景

随着大数据技术的快速发展，Hadoop/YARN 分布式集群在企业级应用中扮演着越来越重要的角色。然而，在实际运行过程中，集群会面临各种类型的故障，如数据倾斜、任务失败、资源瓶颈等。及时准确地检测和诊断这些故障对于保证系统稳定运行至关重要。

本项目旨在构建一个**通用的大数据故障注入和数据收集系统**，为故障检测算法的研究和验证提供实验基础。

### 2. 研究目标

本实验的主要研究目标包括：

#### 2.1 构建故障注入框架

- 实现多种类型的故障注入（数据级、任务级、进程级）
- 支持可配置的故障参数（故障强度、持续时间、触发条件）
- 支持故障组合注入，模拟复杂真实场景

#### 2.2 构建数据收集框架

- 收集故障前后的系统指标（CPU、内存、磁盘、网络）
- 收集集群日志和作业信息
- 收集任务拓扑和执行信息
- 支持自定义时间窗口的数据采集

#### 2.3 故障检测与分类

- 基于收集的数据训练故障检测模型
- 实现多分类算法，识别不同类型的故障
- 评估模型的准确率、召回率、F1值等指标

#### 2.4 故障根因分析

- 分析故障特征与故障类型之间的关联
- 提取关键故障指标和异常模式
- 为故障预警和自愈提供数据支撑

### 3. 实验场景

#### 3.1 故障类型覆盖

本系统支持以下核心故障类型：

| 故障类型 | 故障层次 | 描述 | 严重程度 |
|---------|---------|------|---------|
| **data_bloat** | 数据级 | 数据膨胀故障 | ⭐⭐⭐⭐⭐ |
| **data_skew** | 数据级 | 数据倾斜故障 | ⭐⭐⭐⭐⭐ |
| **task_fail** | 任务级 | 任务失败故障 | ⭐⭐⭐⭐⭐ |
| **long_tail** | 任务级 | 长尾任务故障 | ⭐⭐⭐⭐⭐ |
| **wait_time** | 进程级 | 等待时间异常 | ⭐⭐⭐⭐ |
| **runtime_delta** | 进程级 | 运行时间异常 | ⭐⭐⭐⭐ |
| **exit_time** | 进程级 | 退出时间异常 | ⭐⭐⭐ |

#### 3.2 数据收集场景

每个故障实验包含三个阶段：

1. **故障前阶段（Pre-fault）**
   - 收集故障发生前 3-5 分钟的系统指标
   - 记录正常状态下的基线数据

2. **故障注入阶段（Fault Injection）**
   - 执行故障注入脚本
   - 实时监控故障发展过程

3. **故障后阶段（Post-fault）**
   - 收集故障恢复后 5 分钟的系统指标
   - 记录系统恢复过程

### 4. 预期成果

#### 4.1 数据集

- 构建包含多种故障类型的数据集
- 每个故障类型至少 10 个样本
- 总样本量预计 100+ 条记录

#### 4.2 模型

- 训练多分类故障检测模型
- 实现基于机器学习的故障分类算法
- 模型准确率目标 > 85%

#### 4.3 系统

- 完整的故障注入和数据收集系统
- 可扩展的故障类型支持
- 友好的配置和可视化界面

| 节点名称 | IP地址 | 角色 | 备注 |
|---------|--------|------|------|
| cpf-1 | - | ResourceManager | 主节点 |
| cpf-2 | - | NodeManager | 从节点 |
| cpf-3 | - | NodeManager | 从节点 |
| cpf-4 | - | NodeManager | 从节点 |

#### 1.2 Hadoop 配置

- **HADOOP_HOME**: `/opt/hadoop-3.4.1`
- **HDFS 用户**: `hadoop`
- **YARN 调度器**: Capacity Scheduler

```json
            }
        },
        {
            "type": "task_fail",
            "count": 2,
            "order": 2,
            "enabled": true,

            "count": 1,
            "order": 4,
            "enabled": false,
            "description": "长尾任务故障",
            "params": {

```

#### 2.2 字段说明

| 字段 | 类型 | 说明 |
|-----|------|------|
| `type` | string | 故障类型标识 |
| `count` | int | 故障注入次数 |
| `order` | int | 故障执行顺序 |
| `enabled` | boolean | 是否启用该故障 |
| `description` | string | 故障描述 |
| `params.minutes_before` | int | 故障前数据采集时间（分钟） |
| `params.minutes_after` | int | 故障后数据采集时间（分钟） |
| `params.wait_minutes` | int | 故障注入前等待时间（分钟） |
| `global_settings.output_dir` | string | 测试结果输出目录 |
| `global_settings.random_order` | boolean | 是否随机执行顺序 |
| `global_settings.interval_between_faults` | int | 故障之间的时间间隔（秒） |
| `global_settings.interval_between_runs` | int | 每次运行之间的时间间隔（秒） |

### 4. 数据收集配置

#### 4.1 收集参数

| 参数 | 默认值 | 说明 |
|-----|-------|------|
| DEFAULT_OUTPUT_BASE | "./collected_data" | 默认输出目录 |
| DEFAULT_STEP | 15 | 指标采集间隔（秒） |
| limit | 5000 | 日志查询限制条数 |

#### 4.2 收集模式

系统支持以下数据收集模式：

1. **全部收集 (all)**: 收集日志和指标
2. **只收集日志 (logs)**: 只收集日志文件
3. **只收集指标 (metrics)**: 只收集系统指标
4. **快速收集 (quick)**: 快速收集最近 N 分钟的数据

#### 4.3 使用示例

```bash
# 收集指定时间范围的日志和指标
python3 /scripts/collect_data/collect_data.py all /tmp/output \
  --start "2024-01-18T10:00:00" \
  --end "2024-01-18T10:30:00" \
  --step 15

# 快速收集最近30分钟的数据
python3 /scripts/collect_data/collect_data.py quick /tmp/output --minutes 30
```

### 5. 故障注入脚本配置

#### 5.1 数据膨胀故障 (data_bloat)

**脚本位置**: `/scripts/data_bloat/inject_data_bloat.sh`

- 膨胀倍数：4倍（每行复制4份）
- 输出膨胀数据到 HDFS

#### 5.2 数据倾斜故障 (data_skew)

**脚本位置**: `/scripts/data_skew/inject_data_skew.sh`

- 倾斜键：hot_key（80%数据集中到该键）
- 使用自定义 Partitioner 实现倾斜

#### 5.3 任务失败故障 (task_fail)

**脚本位置**: `/scripts/task_fail/inject_task_fail.sh`

- 失败概率：20%
- 失败类型：RuntimeError

#### 5.4 长尾任务故障 (long_tail)

**脚本位置**: `/scripts/long_tail/inject_long_tail.sh`

- 挂起概率：30%
- 挂起时长：120秒
- 挂起模式：sleep

#### 5.5 等待时间故障 (wait_time)

**脚本位置**: `/scripts/wait_time/inject_wait_time.py`

- 挂起目标：ResourceManager
- 挂起时长：120秒

#### 5.6 运行时间故障 (runtime_delta)

**脚本位置**: `/scripts/run_time/inject_run_time.py`

- 挂起目标：MRAppMaster
- 挂起时长：120秒

#### 5.7 退出时间故障 (exit_time)

**脚本位置**: `/scripts/exit_time/inject_exit_time.py`

- 挂起目标：NodeManager
- 挂起时长：120秒

## 📊 实验流程

### Step 1: 环境准备

1. 启动 Hadoop 集群
2. 确保所有节点 SSH 免密登录配置正确
3. 验证 Prometheus 监控服务正常运行

### Step 2: 配置故障参数

根据实验需求修改 `fault_config.json`：

```bash
vim /scripts/collect_data/fault_config.json
```

### Step 3: 执行故障注入

#### 单个故障测试

```bash
python3 /scripts/collect_data/run_fault_test.py \
  --fault data_skew \
  --output-dir /tmp/fault_test_results \
  --minutes-before 3 \
  --minutes-after 5
```

#### 批量故障测试

```bash
cd /scripts
./start_batch_fault.sh
```

### Step 4: 收集数据

故障注入完成后，系统会自动收集以下数据：

- `/tmp/fault_test_results/<fault_type>/metrics/` - 系统指标
- `/tmp/fault_test_results/<fault_type>/logs/` - 日志文件
- `/tmp/fault_test_results/<fault_type>/collection_summary.txt` - 收集摘要

### Step 5: 数据分析

使用收集的数据进行：

1. 特征工程
2. 模型训练
3. 故障分类
4. 结果评估

## 🔧 常见配置问题

### Q1: 如何修改故障注入次数？

修改 `fault_config.json` 中的 `count` 字段：

```json
{
    "type": "data_bloat",
    "count": 5  // 改为5次
}
```

### Q2: 如何调整数据采集时间窗口？

修改 `params` 中的时间参数：

```json
{
    "params": {
        "minutes_before": 5,   // 故障前5分钟
        "minutes_after": 10,    // 故障后10分钟
        "wait_minutes": 2       // 等待2分钟
    }
}
```

### Q3: 如何添加新的故障类型？

1. 在 `/scripts` 下创建新的故障目录
2. 编写故障注入脚本
3. 在 `fault_config.json` 中添加故障配置
4. 在 `run_fault_test.py` 中添加故障处理逻辑
